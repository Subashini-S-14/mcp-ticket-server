# AI-Ticket MCP Server — Sample Data

This directory contains static, reproducible examples of the inputs and outputs generated by the MCP tools in this repository. These artifacts serve as verifiable proof of functionality for the hackathon judges.

## Directory Structure

* **`inputs/`**: Contains the JSON payloads that the AI Agent (or Claude Desktop) passes as arguments to the MCP Server tools.
* **`outputs/`**: Contains the exact JSON responses returned by the MCP Server for those inputs.

---

## 🛠️ MCP Tools Reference

### 1. `create_ticket`
* **Input** (`inputs/create_ticket_input.json`): Simulates an AI Agent calling the tool to create a new ticket based on a user's natural language complaint about a database timeout.
* **Output** (`outputs/create_ticket_output.json`): Returns the newly created ticket, proving that the server properly generated an ID (`TKT-1a2b3c4d`), assigned default values (`assigned_to: null`, `resolved_at: null`), and recorded ISO-8601 timestamps.

### 2. `get_ticket`
* **Input** (`inputs/get_ticket_input.json`): Simulates the AI Agent requesting the full context of a specific ticket by its ID (`TKT-1a2b3c4d`).
* **Output** (`outputs/get_ticket_output.json`): Returns the ticket details **along with its associated comments** from the database, proving that relational data fetching (Tickets + Comments) works.

### 3. `add_comment`
* **Input** (`inputs/add_comment_input.json`): Simulates the AI Agent appending an update to an ongoing ticket on behalf of a support agent.
* **Output** (`outputs/add_comment_output.json`): Returns the newly generated comment object with its own `CMT-...` ID and timestamp, proving the mutation succeeded.

### 4. `search_kb`
* **Input** (`inputs/search_kb_input.json`): Simulates the AI Agent searching the Knowledge Base for "database timeout replica" before attempting to resolve a ticket.
* **Output** (`outputs/search_kb_output.json`): Returns 5 Knowledge Base articles ranked by relevance score. This proves that the **SQLite FTS5 (Full-Text Search) with BM25 ranking** is functioning correctly and correctly prioritizing articles related to the search terms.

### 5. `list_tickets`
* **Input** (`inputs/list_tickets_input.json`): Simulates the AI Agent filtering the global ticket queue to only show "open", "high/critical" priority "bugs".
* **Output** (`outputs/list_tickets_output.json`): Returns an array of 5 sample tickets that match the criteria. This proves that the dynamic SQL filtering logic in `TicketService` correctly parses and applies the MCP arguments.

### 6. `update_ticket_status`
* **Input** (`inputs/update_ticket_status_input.json`): Simulates the AI Agent updating a ticket's status to "In Progress".
* **Output** (`outputs/update_ticket_status_output.json`): Returns a success message confirming the state transition.

### 7. `resolve_ticket`
* **Input** (`inputs/resolve_ticket_input.json`): Simulates the AI Agent explicitly resolving a ticket after completing work.
* **Output** (`outputs/resolve_ticket_output.json`): Returns a success message confirming the resolution, which triggers a background timestamp injection (`resolved_at`) in the database.

## 🧑‍⚖️ How Judges Can Verify the Results

To verify that these outputs are legitimate and reproducible, judges can run the Python client script which natively invokes all of these tools in sequence:

1. **Ensure dependencies are installed:**
   ```bash
   pip install mcp
   ```

2. **Run the standalone Demo Client:**
   ```bash
   python client.py
   ```

The script will launch the MCP server natively in a subprocess and execute the complete ticket lifecycle flow. You will see the exact JSON structures displayed in this `sample_data` folder printed directly to your terminal.
